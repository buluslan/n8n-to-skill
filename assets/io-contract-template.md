## 填写要点

1. **先填业务目标，再填其他五项**。业务目标写不清楚，触发/参数/产出都是猜的。一句话写不下来，说明还没读懂 workflow。

2. **入口参数和产出形态要具体到文件名+字段**。`tagged_reviews.csv` 优于"输出打标结果"；`tags` 列优于"加个标签字段"。具体的约定能直接驱动 Plan 步设计实现。

3. **质量标准必须可观察**。"洞察质量好"是空话；"洞察报告引用了统计分布的真实数字"是可校验的信号——Verify 步能拿这条对照产出物打勾。

4. **边界条件按"条件 → 行为"写**。每条都是"如果 X，则 Y"，不是"注意 X"。回退路径明确，新 skill 才不会在异常输入上卡死。

5. **触发语义写两端**：原 n8n 触发器是什么（保业务语义连续），新 skill 何时被调用（让 frontmatter 的 description 能命中）。两端对齐=输入对等。

## 用法

Understand 步：复制本表到转换工作目录（如 `transforms/<workflow名>/contract.md`），逐行替换 `{{...}}` 占位和示例列。

Ask 步：把填好的表完整亮给用户，逐行确认或选"用推荐"。

占位全部替换、用户确认后，本表成为 Plan & Build 步的设计输入与新 skill 的验收基线。
