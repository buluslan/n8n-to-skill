## 1. 原始 workflow 概况

**来源文件**：`examples/example-input.json`（亚马逊商品评论 AI 深度分析工作流 V1.0）

**节点构成（18 节点 / 18 条边 / 1 触发器 / 1 循环 / 2 个生成式 AI / 8 个存储协作）**：

| 分类 | 数量 | 节点 |
|---|---|---|
| 1_触发器 | 1 | 表单触发 |
| 2_生成式 AI | 2 | Gemini 评论打标 / Gemini 洞察分析 |
| 5_存储协作 | 8 | 4 × googleSheets + 4 × httpRequest 调 Sheets/Drive API |
| 6_数据转换 | 6 | 5 × code + 1 × set（规范列名 / 解析 AI 结果 / 提取洞察内容 / 准备写入数据 / 验证数据 / 初始化标签列头） |
| 控制流 | 1 | splitInBatches（评论分批循环） |

**业务目标（一句话）**：

> 读入一批亚马逊评论 → AI 给每条评论打 **22 维度标签** → 统计标签分布 → AI 生成 **6 章深度洞察报告** → 输出（带标签的表 + 洞察报告）。

**业务价值**：给跨境卖家做产品迭代决策（改款方向、差评应对、卖点提炼）。

## 2. 完整拓扑图（18 节点 connections 推演）

> 拓扑读法见 `references/workflow-anatomy.md` §3 + §6。这里呈现的是从原 workflow JSON 的 `connections` 字段读出的实际结构。

```
[入口 + 表结构初始化链]                          [循环体]                        [报告生成链]
─────────────────────────                       ──────────                       ─────────────────
表单触发                                         循环处理评论 ──loop(port=1)──→  Gemini评论打标
  │                                               ↑                       │
  └→ 获取表格元数据 ★                             │                       ↓
       └→ 提取SheetId ★                         写回打标结果 ←── 解析AI结果 ←┘
            └→ 扩展列宽至30列 ★                                                    │
                 └→ 初始化标签列头                                                 │ 循环结束(done, port=0)
                      └→ 读取评论数据                                              ↓
                           └→ 验证数据 ──→ 循环处理评论 ──done(port=0)──→ 读取所有已打标数据
                                                                          │
                                                                          ↓
                                                                    统计分析
                                                                          │
                                                                          ↓
                                                                    Gemini洞察分析
                                                                          │
                                                                          ↓
                                                                    提取洞察内容
                                                                          │
                                                                          ↓
                                                                    创建洞察报告表 ★
                                                                          │
                                                                          ↓
                                                                    准备写入数据
                                                                          │
                                                                          ↓
                                                                    写入洞察报告 (end)
```

**★ 标记的 4 个节点**（获取表格元数据 / 提取 SheetId / 扩展列宽至 30 列 / 创建洞察报告表）业务贡献是 **Sheets 表结构管理**——本地 CSV 方案无表结构概念，这些节点的目标贡献被"目标对等"消化掉（判据见 §4）。

### 拓扑读法要点（评论案例踩过的坑）

1. **splitInBatches 端口语义反直觉**：`循环处理评论` 的 port=0 是 **done**（全部批次跑完），port=1 是 **loop**（每批次执行体）。和 IF 的 true/false 直觉相反，必须按 `type` 反查语义。
2. **回环边**：`写回打标结果 → 循环处理评论` 是隐式回环（循环体末端回到 splitInBatches 自身），触发下一批。
3. **executeOnce 标记**：`读取所有已打标数据` 节点带 `executeOnce: true`，表示循环结束后**只读一次**全量打标结果做统计——不随循环重复。

## 3. IO 约定（评论分析具体取值）

> 这是 `references/goal-parity-design.md` §3 五字段方法论 + `references/step-spec.md` §2.2 骨架的实际填法。22 维度 / 6 章的具体枚举来自产物 skill 的 `references/tagging.md` 和 `references/insight.md`。

| 字段 | 内容 |
|---|---|
| **触发语义** | 原：n8n 表单提交（用户在表单里填 asin + 时间范围后启动）。新：CLI 命令 `python3 scripts/io.py count <reviews.csv>` 触发，或 Agent 触发词"分析评论 / 给评论打标签 / 出评论洞察 / 从评论挖卖点 / 评论分析"。 |
| **入口参数** | 必填：评论 CSV 路径（utf-8）。**必须**含评论文本列（`内容` / `内容(翻译)` / `text` 任一）。可选：`标题` / `标题(翻译)` / `星级` / `VP评论`（英文列名 `title` / `rating` / `vp` 同样识别）。 |
| **产出形态** | 3 个文件落 `output/`：① `tagged.csv`（原评论列 + **22 个标签列**）；② `stats.csv`（22 个标签列的值分布，含计数 + 占比）；③ `insight.md`（**6 章深度洞察报告**，禁 Markdown 符号，标题层级用【】包裹）。 |
| **质量标准** | ① 打标覆盖率 100%（每条评论 22 维度全填，无依据填 `未提及` / `不明`，绝不臆造）；② 22 维度枚举值严格按体系（保证统计可比）；③ 洞察报告 6 章齐全（画像/卖点/痛点/改进/机会/典型用户），至少引用 3 个统计支撑；④ 标签分布可复跑稳定（同输入跑两次，top 标签排序一致）。 |
| **边界条件** | ① 评论数 < 5 → 只出 `tagged.csv` + `stats.csv`，跳过洞察并在 `insight.md` 注明"样本不足"；② 单条评论 > 2000 字 → 截断到 2000 字打标，`tagged.csv` 标 `[truncated]`；③ CSV 缺评论文本列 → 报错并指明缺哪列，终止；④ AI 单条打标失败 → 重试 3 次后该条标签全填 `未提及` / `不明`，标 `[TAG_FAILED]` 后继续，不阻塞整体。 |

### 3.2 6 章洞察框架

> 详表见产物 skill 的 `references/insight.md`。

1. 核心用户画像与场景（定量 + 交叉 + 典型画像）
2. 核心卖点与价值验证（满意度归因 + 语义挖掘）
3. 主要痛点与负面归因（问题分布 + 根源追溯 + 严重性评估）
4. 改进建议与优先级（3 条建议，按频率 × 决策影响排高/中/低）
5. 潜在机会与差异化（竞品情报 + 蓝海发现）
6. 典型用户深度解析（1-2 个画像，每画像 1-2 条评论，正负均衡，共 ≥ 4 条）

## 4. 节点 → 能力收敛推演

> 方法学骨架在 `references/goal-parity-design.md` §4 + §5。这里给出评论案例的完整推演表与判据。

### 4.1 8 个存储协作节点 → 3 文件操作 + 4 消化节点

| # | 原节点 | 业务贡献 | 本地 CSV 等价 | 收敛 |
|---|--------|---------|-------------|------|
| 1 | 读取评论数据 (googleSheets.read) | 读入原始评论 | `pd.read_csv('reviews.csv')` | **本地化** |
| 2 | 写回打标结果 (httpRequest→sheets update) | 把标签写回评论行 | 内存改 DataFrame → 写回 CSV | **本地化** |
| 3 | 读取所有已打标数据 (googleSheets.read, executeOnce) | 循环后读全量打标结果做统计 | 内存中 DataFrame 直接复用，免二次读 | **本地化（简化）** |
| 4 | 创建洞察报告表 (httpRequest→sheets create) | 建报告表 | 新建 Markdown 文件 | **本地化** |
| 5 | 写入洞察报告 (googleSheets.append) | 写洞察内容 | 写 Markdown 文件 | **本地化** |
| 6 | 初始化标签列头 (httpRequest→sheets) | 写表头 | `to_csv` 默认 `header=True` 自动写 | **消化（默认满足）** |
| 7 | 获取表格元数据 (httpRequest→drive) | 拿 spreadsheet 的 fileId | 无 | **消化（路径即标识）** |
| 8 | 提取 SheetId (httpRequest→drive) | 拿 sheetId | 无 | **消化（路径即标识）** |
| 9 | 扩展列宽至 30 列 (httpRequest→sheets) | 防写回截断 | 无 | **消化（CSV 列宽自适应）** |

**收敛结果**：8 个存储协作节点（含 1 个扩展列宽节点）→ **3 个文件操作**（读 CSV / 写带标签 CSV / 写洞察 Markdown）+ **1 个免二次读**（内存复用）+ **4 个消化**（表头自动 / fileId / sheetId / 列宽，本地文件无表结构概念）。

### 4.2 4 个消化节点的判据（为什么能消化）

被消化的 4 个节点（#6 初始化列头 / #7 表格元数据 / #8 SheetId / #9 列宽）服务的子目标是 **"为写回 Sheets 做表结构准备"** —— 拿 fileId、拿 sheetId、扩列宽、写表头。本质上是 n8n 为了通过 Sheets API 操作表格而必须完成的"前置准备"。

**两个收敛判据（来自 goal-parity-design §4.2）**：

1. **这组节点服务的子目标是什么？** → 「为写回 Sheets 做表结构准备」
2. **这个子目标能否被一个更简单的实现覆盖？** → 能。本地 CSV 方案下：
   - 文件路径就是标识（不需要 fileId / sheetId）
   - 列宽自适应（不需要手动扩展）
   - 写入用标准库（`to_csv` 默认 `header=True`，不需要单独初始化列头）
   - 不需要 API 调用

两个判据都答得出 → 收敛成立。子目标还在（保证产出可承接），实现被更简单方案覆盖。

> **这是目标对等的正确收敛，不是偷工减料** —— 业务目标（让产出有地方放）还在，实现被更简单的方案覆盖。判据要写进重写方案对照表的"被消化"栏，Verify 阶段有据可查。

### 4.3 2 个 LLM 节点 → 2 次 Claude 调用（不合并）

原 workflow 的 2 个 googleGemini 节点：

- **Gemini 评论打标**：输入评论原文，输出 22 维 JSON 标签
- **Gemini 洞察分析**：输入 22 维标签统计 + 精选样本，输出 6 章洞察报告

迁移为 2 次 Claude 调用，**不合并**——输入依赖不同（一个吃原文，一个吃统计），合并不简化反而增加复杂度。提示词原样保留语义约束（22 维度枚举值 / 6 章框架 / 角色 / 格式约束），剥离平台差异（Gemini 的 `temperature` / `maxOutputTokens` 按 Claude 对应）。

### 4.4 18 节点 → 4 能力的最终收敛

| 原 workflow | 产物 skill |
|---|---|
| 18 节点 / 18 条边 | 4 条核心能力 + 4 步流程 |
| 1 触发器（表单） | description 触发词 + CLI 命令 |
| 8 存储协作节点 | 3 文件操作（+ 4 消化 + 1 免读） |
| 2 Gemini 节点 | 2 次 Claude 原生调用 |
| 6 数据转换节点 | pandas 重写（`df.rename` / `value_counts` / 字典取值等） |
| 1 splitInBatches 循环 | Python `for row in csv` 一行替代 |

## 5. 灰区判据（评论版）

> 判据方法论在 `references/step-spec.md` §4。这里给出评论案例的实际判定。

### 5.1 该转的四条特征（满足 ≥ 3 条就转）

| 判据 | 评论分析的表现 | 结论 |
|---|---|---|
| 可复用 | 用户每次上架新品 / 每周复盘都要跑 | ✅ 强满足 |
| IO 约定稳定 | 评论 CSV → tagged.csv + stats.csv + insight.md，固定 | ✅ 强满足 |
| 以判断/生成为主 | 两次 LLM 调用（22 维打标 + 6 章洞察）是核心 | ✅ 强满足 |
| 本地化代价可控 | Sheets 8 节点 → 3 文件操作，等价达成 | ✅ 满足 |

### 5.2 不该转的四条反特征（命中任何一条就停下反问）

| 反特征 | 评论分析的表现 | 结论 |
|---|---|---|
| 强可审计高频大批量（金融/订单/合规） | 评论分析非审计场景 | ✅ 不触发 |
| 依赖多通道 SaaS 写入 | 原 workflow 只写一个 Sheets | ✅ 不触发 |
| 一次性任务 | 反复跑（每次新品/每周复盘） | ✅ 不触发 |
| 强依赖 cron 定时 / 外部 webhook | 表单触发，非 cron | ✅ 不触发 |

### 5.3 结论

**四条全绿、零反特征 → 转换**。Sheets 节点的"取元数据 / 提取 SheetId / 改列宽"等表结构管理类节点被目标对等消化掉（本地文件无表结构概念），写进重写方案对照表的"被消化"栏。

### 5.4 灰区反例（用于对照）

若评论分析 workflow 末尾还接了"自动发 Slack 日报 + 邮件给老板 + 写周报 Notion"三路分发，则命中反特征 2（多通道 SaaS 写入）——此时在 Ask 步问用户：

- ① 保留三路分发（skill 不再纯粹，要集成 Slack/邮件/Notion SDK）
- ② 只保留核心（评论打标 + 洞察），分发逻辑交给用户手工或另一个 skill

**默认推荐 ②**。

## 6. 产物 skill（amazon-review-analyzer）的 SKILL.md 段落填空示范

> 段落设计逻辑在 `references/output-template.md`。这里给出评论案例的实际填法，对照产物 skill 的 `SKILL.md`。

### 6.1 frontmatter 填法

```yaml
---
name: amazon-review-analyzer
description: 把一批亚马逊商品评论变成「22 维度标签 + 标签分布统计 + 6 章深度洞察报告」，给跨境卖家做产品迭代决策（改款、差评应对、卖点提炼）。当用户提供评论 CSV 路径，并说「分析评论 / 给评论打标签 / 出评论洞察 / 从评论挖卖点 / 评论分析」时，必须使用本 skill。不负责实时采集评论（评论须现成 CSV）；不处理非亚马逊来源评论；不做选品分析（非评论维度）。
allowed-tools: Read, Write, Edit, Bash(python3:*)
license: MIT
metadata:
  source_workflow: 亚马逊商品评论AI深度分析工作流V1.0.json
  goal_alignment: 读评论→AI 22维打标→统计→AI 6章洞察→输出（全链路目标对等，原 18 节点收敛为 4 能力）
  conversion_date: 2026-07-20
---
```

**设计要点**：

- **name** 用业务领域命名（`amazon-review-analyzer`），禁用工具前缀（`n8n-` / `sheets-` / `gemini-`）——原工具在转换中已下线，前缀即 sediment。
- **description 三段式**：① 能力句（做什么 + 业务价值）；② 触发词清单（5 条用户实际话术）；③ 反触发词清单（3 条，每条指向相邻 skill 边界）。≤1024 字符、无裸尖括号。
- **allowed-tools** 精确到子权限：生成式 AI 由 Claude 原生提供（无需 MCP），存储改为本地文件操作 → 只有 `Read` / `Write` / `Edit` / `Bash(python3:*)`。
- **metadata** 三项溯源字段：`source_workflow`（带扩展名）/ `goal_alignment`（一句话对齐）/ `conversion_date`（ISO 格式）。

### 6.2 8 段正文填法要点（每段一行示范）

| 段 | 防的失败模式 | 评论案例填法 |
|---|---|---|
| 1. 定位 | duplication | 给跨境卖家：把一批评论变成「22 维度标签 + 分布统计 + 6 章洞察报告」，支撑产品迭代决策。 |
| 2. 触发 | no-op | 用户给评论 CSV 路径 + 关键词集合（分析/标签/洞察/卖点/评论分析）任一时启动。 |
| 3. 输入 | Agent 优先非问卷 | 评论 CSV，**必须**有评论文本列（`内容` / `内容(翻译)` / `text` 任一）；可选 `标题` / `星级` / `VP评论`。 |
| 4. 输出 | premature completion | 3 个文件落 `output/`：`tagged.csv`（原列 + 22 标签列）/ `stats.csv`（22 标签分布）/ `insight.md`（6 章洞察）。 |
| 5. 核心能力 | duplication + 指引路径 | 4 条 bullet（每条一行）：① 读 CSV 校验（`scripts/io.py count`）；② 22 维打标（Agent，维度见 `references/tagging.md`）；③ 标签统计（`scripts/io.py stats`）；④ 6 章洞察（Agent，框架见 `references/insight.md`）。 |
| 6. 流程 | premature completion | 4 步顺序号 + 每步产物：① 校验 → 打印行数 + 列名；② 打标 → `tagged.csv` + 每 10 条进度；③ 统计 → `stats.csv`；④ 洞察 → `insight.md`。 |
| 7. 凭证边界 | sediment | **无**。原 Gemini key 与 googleSheets OAuth 已消化，LLM 由 Claude 原生提供，存储改本地 CSV，无外部凭证。 |
| 8. 失败降级 | negation + no-op | CSV 缺列 → 报错指明缺哪列；AI 单条失败 → 重试 3 次后填 `未提及` + 标 `[TAG_FAILED]` 继续；< 5 条 → 跳过洞察注明；> 2000 字 → 截断标 `[truncated]`。 |

### 6.3 关键收敛示范

原 n8n **18 节点** → 产物 SKILL.md **4 条核心能力 + 4 步流程**。这是目标对等的最终呈现：节点结构不映射，业务目标对齐即可。核心能力段不写节点清单（"节点 1 做的 / 节点 2 做的"）——那是逐节点翻译的味，违反目标对等。

## 7. 目标对等验收

> 验收方法在 `references/goal-parity-design.md` §6。这里给出评论案例的具体对齐铁证。

### 7.1 同输入双跑比对

准备 **≥ 20 条真实评论**样本（脱敏），两路并行跑：

1. 原 workflow 跑一遍，记录产出（22 维标签表 + 6 章报告）
2. 新 skill 跑一遍，记录产出
3. 逐项比对（用 §7.2 的判据）

### 7.2 对等判据：语义一致，非字面一致

| 维度 | 字面一致（过严） | 语义一致（正确判据） |
|------|----------------|-------------------|
| 22 维度标签 | 每条标签措辞完全相同 | 维度集合完全相同（22 个维度全填，无缺失）；同义枚举值算对齐（"质量好" vs "良好" 都算正面质量） |
| 6 章洞察 | 报告逐字相同 | 6 章齐全（画像/卖点/痛点/改进/机会/典型用户），结论方向一致（都说正面偏多 / 都指同一个主要痛点） |
| 标签分布 | 数字完全相同 | 数量级一致（top 3 标签排序一致，相对比例 ±10% 内） |

### 7.3 22 维度对齐铁证（核心验收项）

**必须排除的差异**（出现即不对等）：

- 22 维度中**任一维度缺失** —— 原 workflow 有"质量_材质"维度，新 skill 没有 → 不对等
- 22 维度中**任一枚举值被自创** —— 原"人群_年龄段"枚举是 18-25/26-35/36-45/46-55/55+/不明，新 skill 加了"未成年" → 不对等（破坏统计可比性）
- 洞察结论方向相反 —— 原 6 章都说"包装破损是主要痛点"新 skill 说"包装完好" → 不对等
- 标签分布数量级悬殊 —— 原 60% 正面新 20% 正面 → 不对等

**允许的差异**：

- 措辞差异（同义表达）
- 顺序差异（标签列表顺序不影响语义）
- 细节展开差异（6 章里的具体引用评论可以不同，但引用的统计支撑必须真实存在）

### 7.4 没有真实输入数据，不能声称成功

**硬规则**：Verify 阶段若无真实输入数据跑通双路比对，交付物里只能说 **"设计已对齐 IO 约定，待真实数据验收"**，不能声称"目标对等已达成"。

理由：LLM 产出有随机性，没有同输入比对就无法证明语义对等。设计阶段的对齐只是"应该对等"，跑过才是"证明对等"。

退而求其次的基准（用户拿不到原 workflow 运行环境时）：

- 用历史产出样本（用户存档的某次运行结果）作为基准
- 或用人工标注的"期望产出"作为基准
- 验收报告里注明基准来源

### 7.5 22 维度 × 6 章 = 完整对齐清单

验收报告必须逐项核：

- [ ] 22 维度全填，枚举值严格按 `references/tagging.md`
- [ ] 无依据维度填 `未提及` / `不明`，无臆造
- [ ] `tagged.csv` 列数 = 原列 + 22 标签列
- [ ] `stats.csv` 覆盖全部 22 维度，每维度有计数 + 占比
- [ ] `insight.md` 6 章齐全，标题用【】包裹（无 Markdown 符号）
- [ ] 洞察报告至少引用 3 个统计支撑（如"包装破损 23 条，占负面 18%"）
- [ ] 同输入双跑，top 标签排序一致
