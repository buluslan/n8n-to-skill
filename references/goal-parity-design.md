# 目标对等设计方法论

> 本文件是 n8n-to-skill 的灵魂文档。Understand 阶段（读懂 workflow 在干什么）和 Verify 阶段（比对产出是否对等）必读。
>
> 本文只讲方法论。节点分类与等价实现判断骨架见 `node-mapping.md`，workflow JSON 静态结构（connections / 表达式 / parameters）见 `workflow-anatomy.md`，脚本接口见 `parse_workflow.py` 头注释。

---

## 1. 目标对等的定义

### 1.1 一句话原则

> **读懂 workflow 在干什么，不翻译它怎么干。**

目标对等对齐的是 **业务目标** 和 **IO 约定**，绝不是节点结构。n8n 的 N 个节点可以收敛成 skill 的 1 个能力，只要最终结果对等。

### 1.2 三条对齐

- **输入对齐**：新 skill 的触发语义与原触发器等价
- **输出对齐**：新 skill 的产出与原 workflow 同等业务价值
- **内部实现自由**：节点之间的拆分、编排、命名都允许自由设计

### 1.3 与逐节点翻译的本质区别

| 维度 | 逐节点翻译 | 目标对等 |
|------|-----------|---------|
| 对齐对象 | 节点结构 | 业务目标 |
| 节点关系 | 1 节点 → 1 函数 | N 节点 → 1 能力（可收敛） |
| 实现绑定 | 保留原实现细节 | 只保留业务结果 |
| workflow 可下线 | 否（仍依赖原 API） | 是 |

**反例（这些做法不是目标对等）：**
- 把 `googleSheets.read` 节点机械翻译成 `read_google_sheet()` 函数 —— 结果：skill 仍依赖 Sheets API，workflow 无法下线
- 保留 `httpRequest` 调 `googleapis.com/drive/v3/files` 拿 SheetId 的实现 —— 这是 n8n 协作 Sheets 的内部细节，不是业务目标

**正例（评论分析案例）：**
原 workflow 用 8 个存储协作节点（4 个 googleSheets + 4 个 httpRequest 调 Google Sheets/Drive API）管理评论数据。目标对等后，新 skill 用 2-3 个本地文件操作（读 CSV / 写带标签的 CSV / 生成报告 Markdown）就实现了同等业务价值。8 → 2-3 的收敛成立，因为业务目标不是"操作 Sheets"，是"读入评论、产出带标签和洞察的分析结果"。

## 2. 提炼业务目标

### 2.1 从 connections 拓扑「走一遍」还原数据流

拿到 workflow JSON，第一步是 **走一遍**，先别急着分类。流程：

1. 找触发器（type 含 `trigger`、`formTrigger`、`webhook` 等）作为起点
2. 顺着 `connections` 里的 `main` 连线往下走
3. 每到一个节点问三个问题：
   - 它 **读入** 了什么？（数据来源）
   - 它 **产出/改变** 了什么？（字段新增/修改/聚合）
   - 它在数据流里扮演什么 **角色**？（搬运 / 计算 / 存储 / 分支）
4. 走到终点（无下游的节点），记录最终产出形态

**评论分析案例走一遍：**

```
触发器（表单提交评论）
  → googleSheets.read（读评论表）
  → 取表元数据 / 提取 SheetId / 扩展列宽至 30 列（3 个节点准备 Sheets 工作区）
  → code 节点（规范列名）
  → splitInBatches（分批循环）
  → googleGemini（给每批评论打标签）
  → code（合并批次结果）
  → googleSheets.append（把标签写回表）
  → code（统计标签分布）
  → googleGemini（生成洞察报告）
  → googleSheets.append（写洞察）
```

走完一遍，业务目标浮现：**读评论 → AI 给每条评论打标签 → 统计 → AI 生成洞察 → 输出（标签 + 洞察报告）**。

走的过程中要刻意忽略 "取 SheetId" 这类动作 —— 它不改变评论数据本身，只是 Sheets 协作的内部准备。这种刻意忽略就是目标对等的第一步。

### 2.2 意图线索三件套

走拓扑遇到看不懂的节点，用这三个线索推断意图（按可信度排序）：

1. **Sticky Note（最可信）**：设计者自己写的旁白，往往直接说"这步是为了 XXX"。位置在 workflow 顶层 `nodes` 里 type 为 `n8n-nodes-base.stickyNote` 的元素，看 `content` 字段。
2. **节点命名（次可信）**：`name` 字段。评论分析里"扩展列宽至30列"这个名字本身就暴露了意图 —— 表结构管理，而非数据处理。
3. **parameters 语义（兜底）**：看参数猜意图。比如 httpRequest 的 url 含 `/drive/v3/files?q=mimeType=spreadsheet` 基本可判定是"拿文件元数据"，属于存储协作的准备工作。

### 2.3 提炼不出就反问

走完拓扑仍说不清业务目标时，**暂停推断**，向用户反问。反问清单：

- "这个 workflow 服务什么业务场景？谁会用它的产出？"
- "如果只能保留一个产出，你希望是什么？（标签表 / 洞察报告 / 还是别的）"
- "触发它的人，最关心它解决什么问题？"

把答案补进业务目标，再继续设计。**反问一次胜过猜错十次。**

## 3. IO 约定（5 字段）

业务目标提炼完后，落到 **IO 约定** 上。约定是验收对等的依据，必须可观察、可验证。五个字段：

| 字段 | 含义 | 评论分析案例 |
|------|------|------------|
| **触发语义** | 什么场景下该跑这个 skill | 用户提交了一批评论（CSV 或表单），想拿到分析结果 |
| **入口参数** | skill 启动时需要什么输入 | 评论数据（CSV 路径或粘贴文本），可选：标签维度、洞察侧重 |
| **产出形态** | skill 完成后给出什么 | 两份产出：① 每条评论带标签的 CSV；② 洞察报告（Markdown 或 CSV） |
| **质量标准** | 产出要达到什么标准 | 标签覆盖每条评论（22 维度全填，无依据填未提及）；洞察报告 6 章齐全（画像/卖点/痛点/改进/机会/典型用户），维度对齐原 workflow |
| **边界条件** | 什么情况会降级或拒绝 | 评论数 < 5 时只出标签不出洞察；单条评论 > 2000 字截断处理；空输入报错退出 |

写约定时的硬要求：
- 每个字段必须 **可验证**（"质量好"不可验证，"标签覆盖每条评论"可验证）
- 边界条件必须 **显式列出**（省了会在 Verify 阶段扯皮）
- 产出形态必须 **可观察**（文件 / 命令行输出 / 截图都行）

## 4. 设计等价实现

### 4.1 决策树：存储类节点 → 本地化 vs 保留集成

存储协作类节点（class 5：googleSheets/notion/airtable/gmail/slack/httpRequest 调协作 API）是收敛的主战场。每个节点都过一遍这个决策树：

```
这个存储节点的业务目的是什么？
├─ A. 纯粹读/写数据（不依赖协作特性）
│   └─ 本地化 → 文件读写 / 本地数据库
├─ B. 依赖协作特性（多人编辑 / 实时同步 / 权限控制）
│   └─ 保留集成 → 用 MCP 或 API 继续对接
└─ C. n8n 为了操作该存储而做的准备工作（取元数据 / 建表 / 改结构）
    └─ 消化 → 本地方案不需要这套准备
```

**判断 B 路径（才保留集成）的三个维度：**
1. **协作**：产出是否需要多人同时编辑？（评论分析产出只给提交者看 → 否）
2. **多端同步**：产出是否要在手机/网页/桌面多端访问？（否）
3. **实时共享**：产出是否需要实时推送给他人？（否）

三个都"否"，走 A 或 C 路径本地化。

**评论分析案例判定：**
- 读评论表 → A，本地化（读 CSV）
- 写回标签 → A，本地化（写 CSV）
- 写洞察报告 → A，本地化（生成报告文件）
- 取表元数据 / 提取 SheetId / 扩展列宽 → C，全部消化（本地文件无表结构概念）

### 4.2 消化冗余节点（N → 1 收敛判据）

判断一组节点能否收敛成一个能力，问两个问题：

1. **这组节点服务的子目标是什么？**（用一句话说清）
2. **这个子目标能否被一个更简单的实现覆盖？**（覆盖即可，不必逐项对应）

两个都答得出，就收敛。

**评论分析案例收敛：**

"取表元数据 / 提取 SheetId / 扩展列宽至 30 列" 这 3 个节点的子目标是 **"为写回 Sheets 做表结构准备"**。本地 CSV 方案没有表结构概念（列宽自适应、无需 SheetId），所以这 3 个节点被一个"写 CSV 文件"的操作整体覆盖。子目标还在（准备能承接产出的地方），实现被更简单方案覆盖。

收敛时要在 IO 约定里记一笔：**"原 workflow 的 X 个表结构节点被本地化方案消化，业务贡献（保证产出可承接）由文件操作默认满足。"** 这样 Verify 阶段有据可查。

### 4.3 LLM 节点 → Claude 原生能力

生成式 AI 类节点（class 2：googleGemini/openAI/langchain/agent）直接用 Claude 原生能力顶替。迁移要点：

1. **提取提示词**：从节点的 `parameters.prompt` 或 `messages` 字段拿出原文
2. **保留语义指令**：分类维度、输出格式、角色设定这类 **语义约束** 必须逐条保留
3. **剥离平台差异**：Gemini 特有的 `temperature`/`maxOutputTokens` 按 Claude 对应（temperature 保留，maxOutputTokens 映射 max_tokens）
4. **合并连续 LLM 节点**：两个 LLM 节点做的事可合并时（比如一个分类一个打分），合成一次调用

**评论分析案例：**
原 workflow 有两个 googleGemini 节点：
- **Gemini-打标签**：输入评论，输出标签（情感/主题/痛点）
- **Gemini-洞察**：输入标签统计，输出洞察报告

迁移后：两个独立 Claude 调用，提示词原样保留语义约束（标签维度、报告结构），合在一个 skill 的两个步骤里执行。**这两个节点不合并** —— 输入依赖不同（一个吃原文一个吃统计），合并不简化反而增加复杂度。

## 5. ★本地 CSV 完整案例：评论分析

### 5.1 业务目标与 IO 约定

- **业务目标**：读入评论 → AI 给每条评论打标签 → 统计标签分布 → AI 生成洞察 → 输出（标签表 + 洞察报告）
- **触发语义**：用户提交一批评论，想拿到带标签和洞察的分析结果
- **入口参数**：评论 CSV（或粘贴文本），可选标签维度
- **产出形态**：① 带标签的 CSV；② 洞察报告 Markdown
- **质量标准**：标签覆盖每条评论（22 维度全填）；洞察报告 6 章齐全（画像/卖点/痛点/改进/机会/典型用户）
- **边界条件**：< 5 条评论只出标签；单条 > 2000 字截断

### 5.2 8 个存储协作节点逐节点推演表

原 workflow 的 8 个存储协作节点（4 个 googleSheets + 4 个 httpRequest 调 Google Sheets/Drive API）逐一对齐：

| # | 原节点 | 业务贡献 | 本地 CSV 等价 | 收敛 |
|---|--------|---------|-------------|------|
| 1 | 读取评论数据 (googleSheets.read) | 读入原始评论 | `pd.read_csv('reviews.csv')` | 本地化 |
| 2 | 写回打标结果 (httpRequest→sheets update) | 把标签写回评论行 | 内存改 DataFrame → 写回 CSV | 本地化 |
| 3 | 读取所有已打标数据 (googleSheets.read) | 循环后读全量打标结果做统计 | 内存中 DataFrame 直接复用，免二次读 | 本地化（简化） |
| 4 | 创建洞察报告表 (httpRequest→sheets create) | 建报告表 | 新建 Markdown 文件 | 本地化 |
| 5 | 写入洞察报告 (googleSheets.append) | 写洞察内容 | 写 Markdown 文件 | 本地化 |
| 6 | 初始化标签列头 (httpRequest→sheets) | 写表头 | `to_csv` 默认 `header=True` 自动写 | 消化（默认满足） |
| 7 | 获取表格元数据 (httpRequest→drive) | 拿 spreadsheet 的 fileId | 无 | 消化（路径即标识） |
| 8 | 扩展列宽至 30 列 (httpRequest→sheets) | 防写回截断 | 无 | 消化（CSV 列宽自适应） |

**收敛结果：** 8 个存储协作节点 → 3 个文件操作（读 CSV / 写带标签 CSV / 写洞察 Markdown）+ 1 个免二次读（内存复用）+ 4 个消化（表头自动 / fileId / sheetId / 列宽，本地文件无表结构概念）。

> 注：原 workflow 另有 6 个**数据转换**节点（5 code + 1 set，归 `6_数据转换` 类）不在本表——抓业务意图用 pandas 重写即可（规范列名→`df.rename`、统计分布→`value_counts`、提取洞察字段→字典取值等），处理思路见 §4.2 与 `node-mapping.md` §6。

### 5.3 重点：被消化的 3 个节点（#2/#3/#4）为什么能消化

这 3 个节点服务的子目标是 **"为写回 Sheets 做表结构准备"** —— 拿 fileId、拿 sheetId、扩列宽。本质上是 n8n 为了通过 Sheets API 操作表格而必须完成的"前置准备"。

本地 CSV 方案下：
- 文件路径就是标识（不需要 fileId/sheetId）
- 列宽自适应（不需要手动扩展）
- 写入用标准库（不需要 API 调用）

这 3 个节点的业务贡献（保证产出可承接）由本地文件操作默认满足，不需要等价动作。**这是目标对等的正确收敛，不是偷工减料** —— 业务目标（让产出有地方放）还在，实现被更简单的方案覆盖。

### 5.4 重要提醒：用户存档的"本地文件版"不作参考

用户归档目录里可能有一份"本地文件版"workflow。**这份文件的节点和 Sheets 版 100% 相同，只是改了文件路径，没有真正本地化。** 本地 CSV 方案是目标对等重写时的 **原创实现**，无现成参考，以本文 §5.2 的推演表为准。

## 6. 目标对等验收方法

设计完不等于成功，必须验收。

### 6.1 同输入双跑比对

准备一份 **真实输入样本**（评论分析的样本：至少 20 条真实评论）。两路并行跑：

1. 原 workflow 跑一遍，记录产出
2. 新 skill 跑一遍，记录产出
3. 逐项比对（用 §6.2 的判据）

### 6.2 对等判据：语义一致，非字面一致

| 维度 | 字面一致（过严） | 语义一致（正确判据） |
|------|----------------|-------------------|
| 标签 | 每条标签措辞完全相同 | 标签集合语义覆盖相同（"质量好" vs "品质优秀" 都算正面质量维度） |
| 洞察 | 报告逐字相同 | 覆盖同维度（情感/痛点/亮点）、结论方向一致（都说正面偏多） |
| 统计 | 数字完全相同 | 数量级一致（标签分布的相对比例一致即可） |

**允许的差异：**
- 措辞差异（同义表达）
- 顺序差异（标签列表顺序不影响语义）
- 细节展开差异（洞察报告里的具体例子可以不同）

**必须排除的差异（出现即不对等）：**
- 标签维度缺失 —— 原 workflow 有"痛点"维度，新 skill 没有
- 洞察结论方向相反 —— 原说"负面偏多"新说"正面偏多"（除非输入样本不同）
- 统计数量级悬殊 —— 原 60% 正面新 20% 正面

### 6.3 没有真实输入数据，不能声称成功

**硬规则：Verify 阶段若无真实输入数据跑通双路比对，交付物里只能说"设计已对齐 IO 约定，待真实数据验收"，不能声称"目标对等已达成"。**

理由：LLM 产出有随机性，没有同输入比对就无法证明语义对等。设计阶段的对齐只是"应该对等"，跑过才是"证明对等"。

如果用户拿不到原 workflow 的运行环境（比如 n8n 已下线），退而求其次：
- 用历史产出样本（用户存档的某次运行结果）作为基准
- 或用人工标注的"期望产出"作为基准
- 验收报告里注明基准来源

## 附：目标对等设计速查清单

设计完一个 skill 前，逐项过一遍：

- [ ] 业务目标用一句话说清了（不依赖节点名）
- [ ] IO 约定 5 字段都填了，每个字段可验证
- [ ] 存储类节点每个都过了决策树（A 本地化 / B 保留集成 / C 消化）
- [ ] 被消化的节点在约定里有记录（消化理由 + 业务贡献如何被覆盖）
- [ ] LLM 节点提示词语义约束逐条保留
- [ ] workflow 可下线（新 skill 不再回调原 workflow 或其依赖的 API）
- [ ] 验收方法已确定（同输入双跑 / 历史样本基准 / 人工标注基准）

全部打勾，才进入 Verify 阶段。
