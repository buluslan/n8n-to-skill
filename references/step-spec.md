# step-spec

> 给 **Ask / Plan 阶段**用的执行约定。把"工作流"落到"状态机 + 可填表格"，让每一步可观察、每一项决策有记录。在 skill-creator 规范基础上补一个【失败/降级路线】字段，形成 5 字段步骤规格。

## 0. 本文件职责

- 给 SKILL.md 第 4 节的五步工作流**补一个【失败/降级路线】列**，形成完整的 5 字段执行规格表。
- 给 IO 约定一张**可直接填的空白表格骨架**（`goal-parity-design.md` 讲"怎么设计"，本文件给"填什么"——不重复）。
- 明确**触发约定（description）的写法**：它是路由合同，不是产品介绍。
- 划清**什么该转 / 什么不该转**的边界。
- 把 Ask 步的反问**展开成五项必问清单**，每项给出问题模板、默认推荐、应答路径。

读这本文件的前提：你已经跑完 Parse，拿到 WorkflowIR；正准备进入 Understand 的尾部或 Ask 的开头。

## 1. 步骤规格表（5 字段）

### 1.1 字段定义

| 字段 | 含义 | 写法约束 |
|---|---|---|
| **进入条件 / 输入** | 这一步启动需要什么 | 列具体产物（文件 / IR / 用户回答），写"无则不能启动" |
| **可执行动作** | 这一步允许做什么 | **禁空动词**（"分析 / 优化 / 处理 / 考虑 / 思考 / 评估"）—— 必须落到可观察动作：跑脚本 / 写文件 / 提问 / 生成文件 / 等待 / 比对 |
| **可观察输出 / 状态** | 跑完后外部能看到什么 | 文件路径 / 终端打印 / 表格 / 用户回答记录——**不能是"完成"二字** |
| **转移条件** | 满足什么才进下一步 | 可布尔判断的条件，不写"差不多 / 基本就绪" |
| **失败 / 降级路线** | 卡住时怎么走 | 三选一：①重试一次；②写 `draft/` + warnings；③硬停止并说明理由 |

### 1.2 五步规格表（在 SKILL.md 第 4 节基础上补"失败/降级"列）

| 步骤 | 进入条件 / 输入 | 可执行动作（必须可观察） | 可观察输出 / 状态 | 转移条件 | 失败 / 降级路线 |
|---|---|---|---|---|---|
| **1 Parse** | n8n JSON / URL / 模板 | 跑 `scripts/parse_workflow.py`；打印节点数、类型分布、凭证引用、子工作流标记 | WorkflowIR JSON + 终端摘要 | IR 有 nodes 且有 connections | JSON 不合法 → 让用户重导出；缺 connections → 标 `draft/` 并停在第 1 步硬停止 |
| **2 Understand** | WorkflowIR | ① 用 `node-mapping.md` 给每个节点归类；② 写业务目标一句话；③ 填 IO 约定表（见本文 §2）；④ 给每类节点写"目标贡献 + 等价实现" | 业务目标陈述 + IO 约定表 + 节点目标贡献图 + 等价实现方案 | 业务目标 + IO 约定 + 等价实现 三项都写成文字 | 节点归类有 `unknown` → 存 `raw-nodes.md` + warnings 继续；业务目标提炼不出 → 直接跳到 §5 反问清单第 1 项 |
| **3 Ask** | §2 的产物 | 把 §5 的五项必问一次性摆给用户，每项附"默认推荐"；等用户逐项回答 | 用户决策记录（五项各一条） | 五项都有明确回答（含"用推荐"） | 用户只答部分 → 把未答项标"用推荐"并继续；用户拒绝所有推荐 → 进硬停止 §1 的第 5 条 |
| **4 Plan & Build** | 决策记录 | ① 先输出 skill 目录树 + 重写方案对照表给用户过目；② 用户点头后才生成 SKILL.md + references + scripts | skill 目录草稿 + 重写方案对照表 | 用户确认方案 + 目录树 | 方案被否 → 回 §2 调整等价实现，重出方案；生成中报错 → 重试 1 次，再失败写 `draft/` + warnings |
| **5 Verify** | 草稿 + 原 workflow + 真实输入 | ① 复用 skill-creator 的 `quick_validate.py` 做硬校验（外部依赖）；② 跑 `check_skill.py` 软校验；③ 同一组真实输入，原 workflow 和新 skill 各跑一遍，比对业务结果 | 校验报告 + `warnings.json` + 目标对等比对记录 | 三层全绿（结构 + 关键段落 + 目标对等） | 结构校验红 → 修后重跑；目标对等不过 → 回 §2 调实现；无真实输入 → 只交 `draft/` + warnings，**禁止声称转换成功** |

### 1.3 动作栏的硬约束

写"可执行动作"时，自检三问：

1. **换一个外行人来看，他能从我写的动词判断这事干没干完吗？** 不能就改。
2. **这个动作跑完，磁盘 / 终端 / 对话里有新东西吗？** 没有就改。
3. **换成另一个 Agent 来执行，他会因为动词太虚而自由发挥吗？** 会就改。

反例 → 正例（示例，以评论分析为例）：

| 反例（空动词） | 正例（可观察） |
|---|---|
| "分析评论 workflow 的业务目标" | "写一句话业务目标 + 填 IO 约定表五字段，贴回对话" |
| "分析 workflow 的业务目标" | "在 IO 约定表里写一句话业务目标，并指出原 workflow 的核心价值环节（判断 / 生成 / 搬运）" |
| "处理外部集成节点" | "把每个外部集成节点在 §2 节点目标贡献图里标一行：节点名 → 业务贡献 → 本地化 / 保留 / 删除" |

## 2. IO 约定表（可直接填的骨架）

### 2.1 空白模板（复制即填）

```markdown
## IO 约定

| 字段 | 内容 |
|---|---|
| **触发语义** | <原 workflow 怎么启动的；新 skill 对应什么触发方式> |
| **入口参数** | <必填参数 + 可选参数，每个写明类型和默认值> |
| **产出形态** | <跑完后生成的文件 / 数据结构 / 终端输出的具体形态> |
| **质量标准** | <可布尔判断的验收条件，如覆盖率 / 字段齐全 / 引用数> |
| **副作用与外部依赖** | <读什么 / 写什么 / 调用什么外部服务 / 依赖什么凭证> |
```

### 2.2 案例：亚马逊评论分析 workflow 的填法

```markdown
## IO 约定

| 字段 | 内容 |
|---|---|
| **触发语义** | 原：n8n 表单提交（用户在表单里填 asin + 时间范围后启动）。新：CLI 命令 `review-analyzer --asin B0xxx --days 30` 或子 skill 触发词"分析这个 asin 的评论"。 |
| **入口参数** | 必填：`asin`（string）、`days`（int，评论时间窗口）。可选：`lang`（评论语言，默认从 asin 站点推断）、`min_reviews`（少于该数报警，默认 10）。 |
| **产出形态** | ① `tagged_reviews.csv`：原评论 + 标签列 + 置信度列；② `tag_distribution.json`：每个标签的条数和占比；③ `insights.md`：AI 洞察报告，含 top 3 痛点 + top 3 卖点 + 各自的占比和原文引用。 |
| **质量标准** | ① 分类覆盖率 ≥ 95%（空标签率 < 5%）；② 洞察报告至少引用 3 个统计支撑（如"<具体数字> 条评论指向同一痛点，占负面 <比例>"）；③ 标签分布可复跑稳定（同输入跑两次，top 3 标签排序一致）。 |
| **副作用与外部依赖** | 读：亚马逊评论（来源由 §5 反问第 4 项决定：已有 CSV / cookie 抓取 / API）。写：本地 `output/` 目录（原 Sheets 已本地化）。外部依赖：LLM 调用（打标 + 洞察），需要 `ANTHROPIC_API_KEY` 或等价凭证。 |
```

### 2.3 与 `goal-parity-design.md` 的分工

| 维度 | 本文件（step-spec） | goal-parity-design.md |
|---|---|---|
| 给什么 | 空白骨架 + 填好的案例 | 设计方法论：怎么从节点反推 IO、怎么判断"目标对等"成立 |
| 何时读 | Ask 步填表 / Plan 步对照 | Understand 步设计等价实现 / Verify 步比对 |
| 重复风险 | 只给表格，不讲"为什么这五字段" | 只讲方法论，不给可填骨架 |

**原则**：本文件不解释字段设计原理，只给"照着填"的模板和案例。设计原理去 `goal-parity-design.md` 查。

## 3. 触发约定：description 是路由合同

### 3.1 公式

frontmatter 的 `description` 是给**路由层**看的合同，不是给用户看的产品介绍。写法：

```
当用户说"《话术 1》 / 《话术 2》 / 《话术 3》"时，必须使用本 skill。
不要用于"《场景 A》"或"《场景 B》"——那是《相邻 skill 名》的职责。
```

三要素：

1. **should-trigger 用用户嘴里的词**，不用技术术语。用户说"每周给我跑一次销售汇总"，不写"执行 ETL pipeline 聚合"。
2. **should-not-trigger 用来挤占位**。列出容易和本 skill 混的相邻场景，明确划界。
3. **长度 ≤ 1024 字符**，无尖括号转义问题（用书名号 `《》` 或引号代替）。

### 3.2 拆解：n8n-to-skill 自己的 description

```
当用户说"把这个 n8n 工作流重写成 skill / 把这个 flow 抽象成可复用能力 / 
分析这个 n8n workflow 并生成 skill / 把 n8n 模板变成 Claude skill / n8n 转 skill"时，必须使用本 skill。
不要用于"从零写一个 n8n 工作流"或"调试 n8n 节点报错"或"让 skill 调用 n8n"——
那是 n8n 构建类 skill 或封装方案的职责。
```

拆解：

- **should-trigger（5 条话术）**：覆盖"重写 / 抽象 / 生成 / 转换 / 转"五个动词 + "n8n workflow / flow / 模板"三个名词的交叉组合。
- **should-not-trigger（3 条反触发）**：① 从零构建（不是转换）；② 调试报错（不是重写）；③ 让 skill 调用 n8n（违背目标对等核心理念）。三条都指向**相邻 skill 的领地**，把模糊地带挤干净。

### 3.3 生成子 skill 时的 description 写法

**通用骨架**：

```
当用户说"《话术 1》 / 《话术 2》 / 《话术 3》"时，必须使用本 skill。
不要用于"《相邻场景 A》"或"《相邻场景 B》"——那是《相邻 skill 名》的职责。
```

**写法要点**：

- should-trigger 用 **原 workflow 触发场景里的用户原话**（从节点备注、表单标签、cron 周期名里挖），不堆同义词。
- should-not-trigger 列出 **容易混淆的相邻场景**，每条指向一个具体的相邻 skill。
- 总长 ≤ 1024 字符，无裸尖括号（`<...>` 要转成引号或书名号）。

**检查清单**：

- [ ] should-trigger ≥ 3 条真实用户话术（不是同义词堆砌）
- [ ] should-not-trigger ≥ 2 条，每条指向一个具体的相邻 skill
- [ ] 总长 ≤ 1024 字符
- [ ] 无裸尖括号
- [ ] 读出声能判断"这个 skill 该不该上"——读不出就重写

> 评论分析子 skill 的完整 description 样例（含话术 / 反触发 / 拆解）见 `examples/example-output/amazon-review-analyzer/CASE_STUDY.md`。

## 4. 边界：什么该转 / 什么不该转

### 4.1 该转成 skill 的四条特征（满足 ≥ 3 条就转）

1. **可复用**：用户会反复手动跑（每月 / 每周 / 每次新品上架），不是一次性任务。
2. **IO 约定稳定**：能写出清晰的"输入 X → 产出 Y"，参数和产出形态不随每次跑而变。
3. **以判断/生成为主**：核心价值是 LLM 的理解 / 分类 / 生成 / 推理，不是机械搬运。
4. **本地化代价可控**：外部集成（电子表格 / 邮件 / 即时通讯 / 文档库）能等价替换成文件 / 命令行，且替换后业务价值不损失。

### 4.2 不该转的四条反特征（命中任何一条就停下反问）

1. **强可审计高频大批量**：金融结算、订单履约、合规日志、财务对账——需要 n8n 的执行日志、重放能力、审计链路。skill 重写后丢审计 → 风险大于收益。
2. **依赖多通道 SaaS 写入且难等价**：同时写 Sheets + Slack + Notion + 邮件 + 钉钉，本地化后失去"多端可达"的业务价值。
3. **一次性任务**：数据迁移、单次清洗、临时报表——直接写脚本即可，包装成 skill 是过度工程。
4. **强依赖 cron 定时 / 外部 webhook 回调**：skill 的触发模型以"用户主动唤起"为主，密集定时和外部回调不是它的强项——保留 n8n 或换专门的调度方案。

### 4.3 灰区决策流程

**判断流程**（任何 workflow 落在灰区都走这三步）：

1. **勾 §4.1 四条特征** —— 满足 ≥ 3 条倾向"转"。
2. **勾 §4.2 四条反特征** —— 命中任何一条倾向"不转"或"拆分"。
3. **两步结论冲突 / 命中反特征但业务上必须转** → 进 Ask 步，把权衡摆给用户（见 §5）。

**通用示例**：

- ✅ **单通道 SaaS + 业务判断为主 → 转**：原 workflow 只写一个外部表（如电子表格），核心是 LLM 分类/生成节点 → 反特征零命中，直接转；SaaS 节点本地化为文件操作即可。
- ⚠️ **多通道分发 + cron 密集 → 保留调度层**：原 workflow 末尾向即时通讯 + 邮件 + 文档库三路分发（命中反特征 2），且按 cron 高频触发（命中反特征 4）。Ask 步问用户：①保留分发（skill 要集成多个 SDK，不再纯粹）；②只保留核心逻辑，分发与调度仍由 n8n 承担，skill 只暴露 CLI。**默认推荐 ②**。

> 评论分析案例的完整判据表（六维度逐条核对 + 灰区反例推演）已迁至 `examples/example-output/amazon-review-analyzer/CASE_STUDY.md`。

## 5. Ask 步反问清单（五项必问）

进入 Ask 步时，把这五项**一次性**摆给用户，每项附"默认推荐"。用户可逐项回答，未答项按默认推荐走。五项全有明确回答（含"用推荐"）才进 Plan & Build。

### 5.1 业务目标对齐

**问题模板**：

> 我读到这个 workflow 的业务目标是："<你写的一句话>"。对吗？还是其实是 <候选 X> / <候选 Y>？

**默认推荐**：Understand 步提炼的那一句话。

**应答路径**：

- 用户确认 → 写进 IO 约定表的"触发语义"字段。
- 用户纠正 → 回 Understand 重填 IO 约定，再回 Ask 继续。
- 用户也说不清 → 硬停止第 5 条，交"待确认"草稿。

### 5.2 凭证提供方式

**问题模板**：

> 这个 workflow 用到这些凭证：<列出>。我打算全部走 `.env.example` 占位 + 运行时读环境变量，明文 key 一律不接收。这样可以吗？

**默认推荐**：全部 `.env.example` 占位。

**应答路径**：

- 用户同意 → 生成 `.env.example`，运行时读环境变量。
- 用户要塞明文 → 拒绝接收（硬停止第 4 条），引导走占位。
- 用户问"凭证从哪搞" → 列每个凭证的获取路径（亚马逊 cookie / LLM API key / Google service account），不代为申请。

**示例（以评论分析为例）**：

> 用到 `ANTHROPIC_API_KEY`（打标 + 洞察 LLM 调用）+ 亚马逊评论源凭证（取决于你选哪种来源）。我会建 `.env.example` 占位两行。这样可以吗？

### 5.3 触发器改造

**问题模板**：

> 原 workflow 的触发器是 `<n8n trigger type>`。新 skill 打算改成 `<CLI 命令 / 子 skill 触发词 / 定时任务>`。这样可以吗？

**默认推荐**：

- n8n 表单 / manual trigger → CLI 命令 + 子 skill 触发词双通道。
- webhook → CLI 命令（外部调）。
- cron → 保留定时（但提示用户：skill 不擅长 cron，可考虑保留 n8n 的调度层）。

**应答路径**：

- 用户同意 → 写进 description 的 should-trigger。
- 用户要保留原触发器 → 评估是否命中反特征 4（cron 密集）；命中则建议保留 n8n 调度层 + skill 做核心逻辑。

**示例（以评论分析为例）**：

> 原 workflow 是表单触发。新 skill 打算改成：CLI 命令 `review-analyzer --asin B0xxx --days 30` + 触发词"分析这个 asin 的评论"。这样可以吗？

### 5.4 存储本地化 vs 保留

**问题模板**：

> 原 workflow 的存储类节点有：<逐个列出>。我打算：<逐个标"本地化 / 保留 / 删除" + 理由>。这样可以吗？

**默认推荐**：

- 单一 Sheets / Notion / 文档库 → 本地化（写本地 CSV / Markdown）。
- 多通道分发 → 只保留核心通道，其他删掉（见 §4.3 灰区反例）。
- 表结构管理类（取元数据 / 改列宽 / 提取 ID） → 删除（被目标对等消化）。

**应答路径**：

- 用户同意 → 写进重写方案对照表。
- 用户要保留某集成 → 评估代价（要引 SDK / 要凭证 / 要测试），代价过高时回 §4.2 反特征 2 重新判边界。

**示例（以评论分析为例）**：

> 原 workflow 的 N 个 Sheets 节点，我打算这样处理：
> - 读评论表 → 本地化（读 `input/reviews.csv`）
> - 写打标结果 → 本地化（写 `output/tagged_reviews.csv`）
> - 写洞察报告 → 本地化（写 `output/insights.md`）
> - 取表格元数据 / 提取 SheetId / 表结构管理类节点 → 删除（本地文件无表结构概念，被目标对等消化）
>
> 这样可以吗？

### 5.5 原 workflow 留作验收标的

**问题模板**：

> Verify 步要做"目标对等比对"：同一组真实输入，原 workflow 和新 skill 各跑一遍，比对业务结果。这需要保留你的原 workflow 能跑 + 给我一组真实输入数据。这样可以吗？

**默认推荐**：保留原 workflow 至交付后用户自行决定下线；用户提供一组脱敏真实输入。

**应答路径**：

- 用户同意 → 进 Plan & Build 时把原 workflow 路径记进 Verify 步。
- 用户已下线原 workflow / 无真实输入 → 只交 `draft/` + warnings，明确标"目标对等未验证"（硬停止第 6 条），**禁止声称转换成功**。
